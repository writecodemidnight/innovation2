


《基于大数据分析的校园社团活动效果评估与资源优化配置系统》需求规格说明书1. 引言1.1 项目背景当前校园社团活动在评估与管理方面存在主观化、数据孤岛化以及资源配置低效等痛点 。个人的看法过于片面，全体的观点又难以总结，缺乏多维度、深层次的客观测评手段 。1.2 项目目标本项目旨在构建一个“数据采集 - 效果评估 - 资源优化”一体化的系统 。通过融合多源数据，运用多种大数据分析与机器学习算法，实现对社团活动质量的科学评估与资源的动态智能调配，最终提升社团活动满意度和资源利用率，推动高校社团管理向数据驱动转变 。2. 用户角色定义系统主要服务于校园内的三类核心用户群体，并为其提供差异化的功能终端：普通学生（学生端）： 参与社团活动的主体，期望便捷地获取活动信息并获得符合自身兴趣的个性化推荐 。社团负责人/组织者（社团端）： 负责策划与执行活动，需要申请活动资源，并渴望获取客观的活动效果反馈以优化后续策划 。校级管理人员（管理端）： 负责全校资源的统筹分配与社团监管，需要全局视角的数据监控和自动化的资源审批决策支持 。3. 核心功能需求3.1 学生端模块活动大厅与个性化推荐： 基于学生兴趣画像系统展示校园活动，利用 K-Means 聚类分析学生活动参与轨迹，生成并推送个性化活动清单 。活动参与记录： 记录学生的活动参与历史与轨迹，展示学生的课余时间分布等数据 。效果评价与反馈中心： 学生可以在活动结束后进行反馈评价，系统支持运用 NLP 情感分析技术对文本反馈进行处理 。3.2 社团端模块活动申报与资源预约： 支持社团在线提交活动策划书，并提交场地、经费、设备等资源的预约申请 。活动效果分析报告： 活动结束后，系统向社团生成多维度雷达图展示的活动得分报告（基于“参与度 - 教育性 - 创新性 - 影响力 - 可持续性”五维评估模型） 。基于数据的策划改进意见： 结合关联规则挖掘（如 Apriori 算法）和计算机视觉（活动现场图片质量分析），为社团提供反向优化的策划建议 。3.3 管理端模块数据监控大屏： 将复杂的数据分析结果以直观的可视化图表展示，实时监控社团活动开展情况、资源配置效果以及学生参与度 。资源池管理： 统一管理全校的场地、经费、设备等社团可用资源，设定资源总量限制等约束条件 。智能决策与资源调度： 系统根据遗传算法生成的资源分配方案，进行自动化/半自动化的资源审批，并支持临时活动资源的插队机制等动态调整 。3.4 数据底座与采集模块（后台服务）多源数据整合： 通过 ETL 工具对接学校教务、财务等 8 个系统，结合学生信息系统获取学业成绩、专业信息等，构建校园社团数据仓库，解决数据孤岛问题 。外部数据抓取： 采用 Python 网络爬虫技术，抓取微信公众号、微博等社团官方社交媒体上的活动动态数据 。4. 核心算法与数据模型需求系统的核心竞争力依赖于以下算法与模型的嵌入支撑：效果评估模型： 需运用层次分析法（AHP）确定指标权重，结合模糊综合评价法生成综合得分，降低人为评估的主观偏差 。资源需求预测模型： 运用时间序列模型（ARIMA/LSTM）和随机森林算法，预测未来（如月度）的场地预约量和经费需求情况 。资源优化配置模型： 以“最大化资源利用率”和“活动效果得分”为目标函数，运用遗传算法（GA）求解多目标优化问题，需支持秒级生成最优配置方案 。多模态数据分析： 需集成 NLP 文本分析以评估活动策划书的创新性，并引入图像识别技术（计算机视觉）进行活动现场图片质量分析 。5. 非功能需求跨平台与轻量化部署： 系统架构需采用 Spring Boot + Vue.js 开发，支持小程序端与 PC 端双端访问，以便于适配高校现有的 IT 环境 。性能要求： 智能优化算法需具备高并发处理能力，保障在活动申报高峰期能快速（秒级）生成资源配置方案 。安全性需求： 在数据收集和分析过程中（如学业成绩、社交关系），必须确保数据的脱敏与安全，保护学生的隐私 。





第一阶段：基建与架构设计（第1个月）
这个阶段的核心是“想清楚再动手”，避免后期因为架构问题推倒重来。

技术栈确认与环境搭建：


后端： 采用 Spring Boot 框架处理业务逻辑和系统接口 。


前端： 采用 Vue.js 架构，准备支持小程序端和 PC 端双端访问的开发环境 。


算法/数据端： 搭建 Python 环境（建议使用 Anaconda），配置 Pandas、Scikit-learn 等基础库 。

数据库与数据仓库设计：

设计关系型数据库（如 MySQL）表结构，涵盖用户表、活动表、资源池等基本业务数据。

规划数据仓库模型，明确如何存储清洗后的高质量数据。

API 接口定义： 前后端团队坐在一起，用 Swagger 或 Apifox 定义好数据交互的接口标准。

第二阶段：数据采集与清洗（第2-3个月）
没有数据，算法就是空壳。这是构建大数据平台的第一场硬仗。

内部数据对接（打破孤岛）：

编写 ETL（提取、转换、加载）脚本，尝试对接学校的教务、财务等 8 个系统，获取学生的学业成绩、专业信息等 。

外部数据爬取：

采用 Python 的网络爬虫技术，抓取社团官方微信公众号、微博等社交媒体上的活动动态数据 。

数据清洗（脏活累活）：

利用 Python 的 Pandas 库，对采集到的数据进行缺失值处理、异常值剔除和重复值删除，统一数据格式和编码规则 。

第三阶段：核心算法与模型攻坚（第4-7个月）
这是本项目“含金量”最高，也是你们团队在申请书中提到可能存在技术短板  的环节。建议分工并行开发：

模型 1：学生活动画像系统

使用 K-Means 聚类分析学生的活动参与轨迹，将学生划分为不同的兴趣群体，为后续个性化推荐做准备 。

模型 2：五维活动效果评估体系

基于“参与度 - 教育性 - 创新性 - 影响力 - 可持续性”构建模型 。

运用层次分析法（AHP）确定指标权重，结合模糊综合评价法计算得分 。


进阶挑战： 引入 NLP 情感分析处理学生的文本反馈，以及使用图像识别技术（计算机视觉）对活动现场图片质量进行量化分析 。

模型 3：资源需求预测模型

基于时间序列分析算法（如 ARIMA/LSTM 模型）和机器学习算法（如随机森林、神经网络），预测未来场地、资金等资源的需求 。

使用 Apriori 算法挖掘“活动类型 - 资源消耗”的关联规则 。

模型 4：智能资源调度算法

编写遗传算法（GA），以“最大化资源利用率 + 活动效果得分”为目标函数，在资源总量和时间冲突的约束下，求解最优的资源分配方案 。

第四阶段：前后端业务系统开发（第8-9个月）
在算法模型跑通（哪怕是用脚本能输出结果）后，开始把它们封装进可视化的系统中。

后端接口开发 (Spring Boot)：

实现用户鉴权、活动审批流、资源增删改查等基础业务。

编写接口调用 Python 算法服务，实现业务逻辑与算法结果的串联。

前端页面开发 (Vue.js / 小程序)：

学生端： 开发活动推荐流、报名接口、评价反馈页面。


社团端： 开发活动申报表单、效果分析报告（引入 Echarts/AntV 绘制多维度雷达图） 。


管理端： 开发资源池管理界面和数据监控大屏 。

第五阶段：系统集成、测试与部署（第10个月）
把散落的零件组装成一台跑车，并确保它上路不会散架。

系统集成： 确保前端能正确显示后端传来的数据，后端能顺利调用 Python 的算法模型并返回结果。

全链路测试：

进行单元测试、集成测试，修复系统漏洞和功能缺陷 。

特别关注并发测试：如果在社团招新季大家同时抢资源，遗传算法求解是否会卡死服务器？


上线部署： 将系统部署至云服务器（如阿里云、腾讯云），并进行压力测试和性能优化 。

第六阶段：试运行与闭环优化（第11-12个月）
系统做出来了，得让大家用起来，并在实战中打磨。

小范围试运行： 挑选几个活跃的社团作为“种子用户”，让他们用系统走一遍完整的“活动申报 - 资源调度 - 效果评估”流程。

收集反馈与迭代：

根据社团管理部门和学生的真实反馈，调整系统的交互细节。

根据实际产生的数据，重新训练和微调你的预测模型和评估权重，形成“策划 - 执行 - 评估 - 改进”的全流程数字化闭环 。


成果沉淀： 整理项目各阶段文档，撰写学术论文，申请软件著作权等结题材料 。















# 校园社团活动评估系统 - 后端架构设计文档

## 项目概述

### 1.1 项目名称
基于大数据分析的校园社团活动效果评估与资源优化配置系统

### 1.2 项目目标
构建一个"数据采集 - 效果评估 - 资源优化"的一体化系统，通过整合多源数据并运用大数据分析和人工智能算法，客观量化活动质量，实现校园资源的智能调度与优化配置。

### 1.3 核心需求
- **学生端**：个性化活动推荐、活动报名与参与记录、活动效果评价
- **社团端**：活动申报与策划书提交、资源预约申请、效果分析报告查询
- **管理端**：多源数据监控大屏、资源池管理、智能审批与决策
- **后台算法**：多源数据融合、五维评估模型、资源需求预测与优化调度

## 技术栈选型

### 2.1 后端技术栈
- **核心框架**：Spring Boot 3.2.x (Java 21)
- **构建工具**：Maven 3.9+ 或 Gradle 8.5+
- **关键依赖**：
  - Spring Security (JWT认证 + OAuth2客户端)
  - Spring Data JPA (数据库ORM)
  - Spring Boot Actuator (健康检查)
  - MapStruct (对象映射)
  - Lombok (减少样板代码)

### 2.2 算法服务
- **核心框架**：Python 3.10+ + FastAPI 0.104+
- **算法库**：
  - scikit-learn (K-Means聚类、AHP层次分析)
  - pandas/numpy (数据处理)
  - tensorflow/pytorch (LSTM时间序列预测)
  - OpenCV (计算机视觉分析)
  - nltk/spaCy (NLP情感分析)

### 2.3 数据存储
- **主数据库**：PostgreSQL 15.x (支持JSONB和窗口函数)
- **缓存层**：Redis 7.2.x (会话管理、热点数据缓存)
- **对象存储**：阿里云OSS (文件存储)

### 2.4 基础设施
- **容器化**：Docker 24.x + Docker Compose v2.23+
- **云服务**：阿里云ECS (4核8G Ubuntu 22.04)
- **CI/CD**：GitHub Actions 或 Gitee Go
- **监控**：Spring Boot Admin + Prometheus + Grafana

### 3.2 架构优势
1. **适合小团队**：5人团队可快速上手，减少分布式系统复杂性
2. **技术栈隔离**：Java擅长业务逻辑，Python擅长算法，各取所长
3. **渐进式演进**：未来如需拆分，可从单体中逐步剥离服务
4. **成本可控**：基础设施简单，云服务器成本可控

## 模块划分

### 4.1 Java核心业务单体模块

#### 4.1.1 用户模块 (`user-service`)
- **职责**：用户注册、登录、个人信息管理、权限控制
- **核心类**：`UserController`, `UserService`, `UserRepository`, `WechatAuthService`
- **数据库表**：`users`, `user_roles`, `permissions`

#### 4.1.2 活动模块 (`activity-service`)
- **职责**：活动创建、审批、查询、报名、记录
- **核心类**：`ActivityController`, `ActivityService`, `ActivityRepository`, `ActivityRecommendationService`
- **数据库表**：`activities`, `activity_participants`, `activity_resources`

#### 4.1.3 资源模块 (`resource-service`)
- **职责**：场地、设备、经费的资源池管理、预约、分配
- **核心类**：`ResourceController`, `ResourceService`, `ResourceOptimizationService`, `ResourceRepository`
- **数据库表**：`resources`, `resource_reservations`, `resource_budgets`

#### 4.1.4 评估模块 (`evaluation-service`)
- **职责**：收集评价、计算基础指标、生成报告
- **核心类**：`EvaluationController`, `EvaluationService`, `ReportGeneratorService`, `EvaluationRepository`
- **数据库表**：`evaluations`, `evaluation_metrics`, `activity_reports`

#### 4.1.5 管理模块 (`admin-service`)
- **职责**：数据看板、审批流程、系统配置
- **核心类**：`AdminController`, `DashboardService`, `ApprovalWorkflowService`, `SystemConfigService`
- **数据库表**：`approval_records`, `system_configs`, `operation_logs`

### 4.2 Python算法服务API设计

#### 4.2.1 基础信息
- **服务URL**：`http://algorithm-service:8000/api/v1`
- **通信协议**：REST API (JSON格式)
- **超时设置**：默认30秒
- **重试策略**：指数退避，最多3次

#### 4.2.2 算法接口

| 端点 | 方法 | 输入 | 输出 | 用途 |
|------|------|------|------|------|
| `/recommend/kmeans` | POST | `{user_id, history_activities, top_n}` | `{recommended_activities: [...]}` | K-Means个性化推荐 |
| `/evaluation/ahp` | POST | `{activity_id, metrics_data, weights}` | `{scores: {...}, radar_data: {...}}` | AHP层次分析法评估 |
| `/nlp/sentiment` | POST | `{texts: ["text1", "text2"]}` | `{sentiments: [score1, score2]}` | NLP情感分析 |
| `/prediction/lstm` | POST | `{historical_demand, time_features}` | `{predicted_demand: [...]}` | LSTM资源需求预测 |
| `/optimization/ga` | POST | `{activities, resources, constraints}` | `{allocation_plan: {...}}` | 遗传算法资源优化 |
| `/mining/apriori` | POST | `{transactions: [...]}` | `{rules: [...]}` | Apriori关联规则挖掘 |
| `/cv/quality` | POST | `{image_urls: [...]}` | `{quality_scores: [...]}` | CV活动现场图片质量分析 |

## 数据库设计

### 5.1 核心实体关系

#### 5.1.1 用户与角色（RBAC）
```sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    openid VARCHAR(100) UNIQUE,  -- 微信OpenID
    username VARCHAR(50) NOT NULL,
    avatar_url VARCHAR(255),
    email VARCHAR(100),
    phone VARCHAR(20),
    role VARCHAR(20) DEFAULT 'STUDENT', -- STUDENT, CLUB_ADMIN, SYS_ADMIN
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### 5.1.2 社团管理
```sql
CREATE TABLE clubs (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    description TEXT,
    category VARCHAR(50), -- 学术、文艺、体育、公益等
    logo_url VARCHAR(255),
    president_id BIGINT REFERENCES users(id),
    faculty_advisor VARCHAR(100),
    status VARCHAR(20) DEFAULT 'ACTIVE', -- ACTIVE, INACTIVE, SUSPENDED
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### 5.1.3 活动相关
```sql
CREATE TABLE activities (
    id BIGSERIAL PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    description TEXT,
    activity_type VARCHAR(50), -- lecture, workshop, competition, social
    status VARCHAR(20) DEFAULT 'DRAFT', -- DRAFT, PENDING, APPROVED, REJECTED, COMPLETED
    start_time TIMESTAMP NOT NULL,
    end_time TIMESTAMP NOT NULL,
    location VARCHAR(200),
    capacity INT,
    current_participants INT DEFAULT 0,
    club_id BIGINT REFERENCES clubs(id),
    created_by BIGINT REFERENCES users(id),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    CHECK (end_time > start_time)
);
```

#### 5.1.4 资源管理
```sql
CREATE TABLE resources (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    type VARCHAR(50) NOT NULL, -- VENUE, EQUIPMENT, BUDGET
    capacity INT,
    unit VARCHAR(20), -- 单位：人、个、元
    available_count INT DEFAULT 0,
    total_count INT NOT NULL,
    constraints JSONB, -- 约束条件，如时间限制、使用规则
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### 5.1.5 五维评估指标
```sql

    activity_id BIGINT PRIMARY KEY REFERENCES activities(id),
```

## 安全设计

### 6.1 认证体系
- **JWT Token**：采用HS256算法，有效期2小时
- **Refresh Token**：有效期7天，用于获取新Access Token
- **Token黑名单**：Redis存储已注销但未过期的Token

### 6.2 微信OAuth2.0集成
```
1. 前端重定向 → 微信授权页面
2. 用户授权 → 返回code
3. 后端用code + appsecret → 换取access_token和openid
4. 验证openid → 生成JWT返回前端
5. 前端存储JWT → 后续请求携带Authorization头
```

### 6.3 RBAC权限模型
- **角色层级**：SUPER_ADMIN, DEPARTMENT_ADMIN, CLUB_ADMIN, STUDENT
- **权限粒度**：activity:create/read/update/delete/approve, resource:reserve/allocate/manage, user:manage/assign_role, evaluation:submit/view_report

### 6.4 API安全防护
- **SQL注入防护**：使用JPA参数化查询
- **XSS防护**：输入输出过滤，Content-Type检查
- **CSRF防护**：前后端分离架构天然防御
- **限流防刷**：Redis实现接口限流（如登录接口60次/小时）
- **敏感数据脱敏**：返回数据时屏蔽手机号、邮箱等敏感信息

- **版本控制**: 时间戳版本，支持回滚
- **内存管理**: 监控内存使用，自动清理旧模型
